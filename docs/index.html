01Mission

Build a canonical, provenance-rich catalog of complex systems, metrics, and observations across scientific domains — biological, ecological, technological, social, physical, computational, economic, cognitive — and surface cross-domain structural and dynamical patterns.

The catalog is a shared substrate: every entry is a structured record with citations, every claim resolves to evidence, and every derivation from the registry is reproducible from a single git commit.

02Architecture in one paragraph

The filesystem is the source of truth. registry/ holds canonical entities (systems, metrics, sources, observations) as YAML and JSONL. taxonomy/ holds controlled vocabularies. schemas/ holds JSON Schema Draft 2020-12 definitions that every canonical record must validate against. ops/ holds operational state for agents: a directory-based task queue, run reports, and event logs. warehouse/ holds derived Parquet and DuckDB that are never hand-edited — always regenerable from registry/. releases/ holds publishable snapshots described by Data Package and RO-Crate metadata.

Three invariants. (1) The filesystem is canonical; the warehouse is derived. (2) Every canonical record validates against a JSON Schema. (3) Every numeric claim has at least one resolvable evidence reference.

04Schemas and validation

Seven JSON Schema Draft 2020-12 files constrain every structured record in the repo. The coc validate command runs a two-pass check: shape (against the schema) and taxonomy resolution (every taxonomy_refs entry and metric family must resolve against taxonomy/source/*.yaml).

SchemaCoversID format
systemA complex system with boundary, components, interaction types, scalessys-NNNNNN--slug
metricA metric definition with unit, scale of measurement, applicabilitymtr-NNNNNN--slug
sourceA literature source (paper, book, dataset)src-NNNNNN--slug
observationA (system, metric) value with evidence and confidenceobs-<8-hex>
taskA unit of work in the directory-based queuetsk-YYYYMMDD-NNNNNN
runA run report (per-task execution summary)run-<ulid>
eventAn event-log entry (task lifecycle, provenance, runs)ev-<ulid>

The same validator is invoked by the git pre-commit hook, by the validate GitHub workflow, and by the review-records skill. A record that fails validation cannot land in registry/.

05Taxonomy

Four hand-curated controlled vocabularies anchor the catalog. Every canonical record must reference them by slug; ad-hoc strings fail validation.

  • system-domains — ecological, biological, technological, social, physical, computational, economic, cognitive.
  • system-classes — structural archetypes informed by Boulding's hierarchy (framework, clockwork, cybernetic, open, biological, social, symbolic, transcendent, plus engineered extensions).
  • metric-families — boundary-scale, composition, topology, dynamics, information, feedback-adaptation, throughput, resilience.
  • evidence-types — direct, derived, proxy, simulation, expert_estimate.

coc export-taxonomy emits a SKOS Turtle file and a flat JSON label map under taxonomy/exports/. The current bootstrap graph has 202 triples.

06Agent operating model

Two agent runtimes — Claude Code and OpenAI Codex — operate in this repo under a single shared contract. The AGENTS.md file is the canonical operating model; CLAUDE.md and .codex/config.toml are one-liner pointers.

Three-layer prompt stack

Every task execution reads three files, in order:

  1. Layer 1 — AGENTS.md: durable rules, applies every time (quality bar, lifecycle, sensitive actions).
  2. Layer 2 — skills/<name>/SKILL.md: workflow-specific preconditions, procedure, output shape, block conditions.
  3. Layer 3 — task manifest: the concrete job — inputs, output_targets, and acceptance tests for this single run.

The task manifest overrides skill defaults; the skill overrides AGENTS.md silence. Each layer's violations block at a different scope: manifest-level at task close, skill-level at review, AGENTS-level at the pre-commit / CI boundary.

Operating loop

  1. Read the task manifest (ops/tasks/leased/<task-id>.yaml).
  2. Read the active skill (skills/<skill>/SKILL.md).
  3. Read the schemas referenced by the task's acceptance_tests.
  4. Read only the source records the task references.
  5. Produce the declared outputs.
  6. Run uv run coc validate on anything structured.
  7. Append a run report and event; call uv run coc complete <task-id>.

07Task lifecycle

The queue is a set of directories under ops/tasks/. The state of a task lives in two places — the state: field inside the YAML and the directory it sits in — and these must agree. Every transition keeps them in sync. Atomic os.rename across directories on the same filesystem is the lock: two agents racing to claim the same task see exactly one winner.

A lease carries a TTL (default 60–90 minutes). The coc requeue janitor returns stale leases to ready/ up to max_attempts; beyond that they move to failed/. Every transition appends a line to ops/events/task-events.jsonl.

08Quality bar

These rules are enforced by the validator, not by convention. A record that breaks any of them cannot be committed.

  • No uncited numeric claims in canonical records.
  • Every observation includes value_kind, confidence, and at least one evidence_ref that resolves against registry/sources/*/evidence.jsonl.
  • Every metric declares applicability conditions.
  • Every system includes boundary, components, interaction_types, and scales.
  • All taxonomy references (system-domain:*, system-class:*, metric-family:*) resolve against taxonomy/source/*.yaml.
  • Nothing under registry/sources/*/raw/ is ever edited.
  • Nothing under warehouse/ is hand-edited.

Sensitive actions

Agents must request explicit approval before:

  • Deleting accepted records under registry/.
  • Overwriting canonical metric definitions.
  • Modifying anything under registry/sources/*/raw/.
  • Publishing to releases/.
  • Bulk edits spanning multiple systems.
  • Pushing to git remote or opening a PR.

09The coc CLI

A single Click-based entrypoint (src/coc/cli.py, registered as coc via pyproject.toml). All long-running work goes through it so behaviour is identical between interactive sessions and CI.

CommandPurpose
coc validate [path]Schema shape + taxonomy resolution across the full repo.
coc lease <task-id>Atomically move ready/ → leased/.
coc heartbeat <task-id>Refresh lease liveness.
coc complete <task-id> --state <review|done|blocked|failed>Finalize a task and append a lifecycle event.
coc requeueJanitor — return expired leases to ready/.
coc materializeRebuild warehouse/parquet/* and the DuckDB view from the registry.
coc release [--date YYYY-MM-DD]Build a releases/snapshot-*/ with Data Package + RO-Crate + manifest.
coc export-taxonomyRegenerate SKOS Turtle and labels.json.
coc eval [skill]Run QC evals over observations and goldens over each active skill.

Claude Code slash commands under .claude/commands/ — such as /coc-validate, /coc-materialize, /coc-release, /coc-new-task — wrap the same invocations.

10Skill roster

Each skill is a self-contained directory with a SKILL.md manifest (frontmatter + markdown body). Seven are active; one is scaffolded-but-disabled until registry breadth warrants it.

SkillPurposeStatus
setup-repoBootstrap or repair repository scaffolding.active
scout-systemsDiscover candidate systems and candidate metrics.active
define-metricsCurate metric definitions, rubrics, applicability rules.active
profile-systemDefine a system's boundary, components, scales, taxonomy.active
extract-observationsFill system–metric values with evidence and confidence.active
review-recordsSchema, provenance, citation, methodological checks.active
materialize-warehouseRebuild Parquet + DuckDB + release snapshots.active
analyze-archetypesClustering, graph building, hypothesis generation.disabled

analyze-archetypes unlocks once the registry contains ≥200 validated observations spanning ≥5 system classes with ≥3 observations each. The gate is encoded in its SKILL.md preconditions.

11QC: fixtures, evals, goldens

Fixtures

Every schema has a matched pair of fixtures under qc/fixtures/{valid,invalid}/. The validator is parametrized over them: valid fixtures must pass, invalid fixtures must fail with the expected reason. This catches regressions in the schema itself, not just in records.

Evals (declarative)

YAML files under qc/evals/ define checks as Python expressions evaluated in a sandbox over the observation corpus. Two ship out of the box:

  • minimum-one-evidence-ref — every observation has ≥1 evidence ref that resolves.
  • no-uncited-numeric-claims — every numeric value is backed by at least one resolvable evidence reference.

Goldens (per skill)

Each active skill owns at least one qc/goldens/<skill>/<scenario>/ directory containing an input.yaml (a schema-valid task manifest) and expected.yaml (assertions). coc eval enforces that every active skill has at least one passing golden; the GitHub validate workflow fails the PR if this drops.

12Release packaging

coc release re-materializes the warehouse, copies the Parquet tables into releases/snapshot-YYYY-MM-DD/data/, and emits three metadata files:

  • datapackage.json — Frictionless Data Package descriptor with per-resource SHA-256 and CC-BY-4.0 license.
  • ro-crate-metadata.json — RO-Crate metadata for researcher-facing discovery.
  • manifest.md — human-readable index with row counts and per-file hashes.

A snapshot is regenerable by definition: given the registry at the referenced commit, coc release produces the same bytes modulo timestamps. frictionless validate releases/snapshot-*/datapackage.json currently reports 7/7 tables VALID.

13GitHub integration

Three workflows, all in .github/workflows/:

  • validate.yml — on every push and PR. Runs coc validate, scripts/lint_skills.py, and coc eval. The PR-level quality gate.
  • ci.yml — ruff (check + format), mypy on src/, pytest matrix on Python 3.11 and 3.12.
  • materialize.yml — weekly cron (Mondays 06:00 UTC) plus manual dispatch. Rebuilds the warehouse, builds a release, validates the Data Package descriptor, and uploads the snapshot as a workflow artifact. Confirms the registry still regenerates cleanly.

Issue templates under .github/ISSUE_TEMPLATE/ (new-system, new-metric, bug) are GitHub forms with fields that mirror the schema requirements — a proposer can't submit a system without declaring boundary, components, and scales. The PR template embeds the AGENTS.md quality-bar checklist.

CODEOWNERS protects registry/, taxonomy/source/, schemas/, and the agent operating-model files. Branch-protection rules are documented in .github/BRANCH_PROTECTION.md with a copy-paste gh api command to apply them.

14Getting started

Requires Python 3.11+ and uv.

git clone https://github.com/shiftynick/catalog-of-complexity.git
cd catalog-of-complexity
uv sync --extra dev
uv run coc --help

# sanity-check: should all pass from a fresh clone
uv run coc validate
uv run pytest
uv run coc eval

# try the dry-run lifecycle
uv run coc lease tsk-20260422-000001   # already in done/, just for shape
uv run coc materialize
uv run coc release

Opening in an agent

  • Claude Code: opens the repo, reads CLAUDE.md (which points at AGENTS.md), loads slash commands from .claude/commands/, and respects permissions from .claude/settings.json.
  • OpenAI Codex: loads .codex/config.toml, which declares approval policies for mutating writes and allowlists read-only commands.

Contributing a new system or metric

  1. Open a Propose a new system or Propose a new metric issue using the GitHub form.
  2. A reviewer converts the issue into a task manifest under ops/tasks/ready/ with the matching skill.
  3. An agent leases the task, produces the canonical record, validates it, and moves the task to review/.
  4. Reviewer approves; task moves to done/; warehouse re-materializes on the next materialize.yml run.

15What's next

Bootstrap is complete (phases 0 through 12). Next, the catalog gets populated. Near-term priorities:

  • Extend the taxonomy and metric-family coverage as real systems expose gaps.
  • Ingest a first batch of high-quality sources and generate observations for a representative system from each domain.
  • Once analyze-archetypes preconditions are met (≥200 validated observations across ≥5 system classes), enable it and start producing cross-domain pattern reports.
  • Publish the first tagged release snapshot under releases/.

Everything after bootstrap is routine catalog work: propose → task → skill run → review → commit. The machinery is in place; the interesting science starts now.
